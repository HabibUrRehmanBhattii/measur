# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

**measur** — a Next.js 15 (App Router) measurement submission system for custom racing suits/helmets. Customers submit body measurements → data lands in Google Sheets + confirmation email fires. An admin panel (cookie-auth) views and stats the submissions. Deployed on Vercel (`measur-gilt.vercel.app`). UI is in German.

## Commands

```bash
npm install              # install deps (required before first dev run)
npm run dev              # dev server with Turbopack on http://localhost:3000
npm run build            # production build (ESLint + TS errors are IGNORED — see next.config.ts)
npm run start            # run the built app
npm run lint             # ESLint (next/core-web-vitals + next/typescript)

npm run e2e              # E2E headless (auto-starts dev server)

```

Create `.env.local` (never committed — see .gitignore):

```bash
# Google Sheets
GOOGLE_SPREADSHEET_ID=...
GOOGLE_CLIENT_EMAIL=...            # service account email
GOOGLE_PRIVATE_KEY=..."            # multi-line; the code replaces \\n with real newlines

# Email (Gmail app password)
EMAIL_USER=...
EMAIL_APP_PASSWORD=...
EMAIL_RECIPIENT=...                # where notification emails go

# Admin auth
ADMIN_USERNAME=...
ADMIN_PASSWORD_HASH=...            # SHA-256 hex of the password (generated by setup-admin)
```

## Architecture

### Data flow (the big picture)

The database is **Google Sheets** (`Sheet1!A:F`), not a traditional DB. These utilities are the backbone:

- **`app/utils/sheets.ts`** — a `SheetsConnectionPool` singleton (5 reusable `sheets_v4.Sheets` clients) + a `withSheetsClient(fn)` wrapper that checks out a client, runs your query, and releases it. All Sheets reads/writes go through this pool.
- **`app/utils/email.ts`** — Nodemailer Gmail transport with connection pooling (`maxConnections: 3`, `rateLimit: 5`). `sendMeasurementEmail(data)` builds an HTML receipt from a template literal and sends it.
- **`app/utils/cache.ts`** — `ApiCache` singleton: in-memory `Map` with TTL + a `getOrSet(key, fn, ttl)` pattern. Used by `/api/admin/stats` (5-min TTL) and invalidated by `apiCache.delete('admin-stats')` after each new submission.

### API routes

| Route | Purpose | Notes |
|-------|---------|-------|
| `POST /api/submitMeasurement` | Public form submission | Validates → checks duplicate order# via `withSheetsClient` → appends row → formats row wrap/alignment → sends email → invalidates stats cache |
| `POST /api/admin/login` | Admin login | `validateCredentials()` compares SHA-256 hash; sets `admin_auth_token` + `admin_token_expiry` httpOnly cookies via `setAuthCookie()`. Returns generic "Invalid username or password" to prevent enumeration. |
| `DELETE /api/admin/login` | Logout | Clears auth cookies |
| `POST /api/admin/logout` | Alt logout | Same as above, different HTTP verb |
| `GET /api/admin/stats` | Dashboard stats | `revalidate = 60`, 5-min `apiCache`, returns total/pending/completed counts. Status column (index 5) holds `Pending`/`Completed`. |
| `GET /api/admin/measurements` | All submissions | `force-dynamic`, reads `Sheet1!A1:F`, skips header row, maps to `{orderNumber, ebayUsername, type, measurements, date, status}` |

All routes return structured `NextResponse.json()` with `success`/`error` fields.

### Auth flow

1. **`scripts/setup-admin.js`** hashes a password with SHA-256 and writes `ADMIN_USERNAME` + `ADMIN_PASSWORD_HASH` to `.env.local`.
2. Login POST → `validateCredentials()` re-hashes and compares → issues token cookies.
3. **`middleware.ts`** (Edge) intercepts `/admin/*` and `/api/admin/*`: reads `admin_auth_token` + `admin_token_expiry` cookies; redirects to `/admin-login` if missing or expired. `/admin-login` and `/api/*` are explicitly bypassed.
4. **`app/utils/auth.ts`** provides the cookie helpers (`setAuthCookie`, `clearAuthCookies`, `isAuthenticated`, `requireAuth`) used by route handlers. Cookies are `httpOnly`, `sameSite: strict`, `secure` in production, 24h expiry.

### Theme system

Custom (not Tailwind dark mode). **`app/layout.tsx`** creates a `ThemeContext` providing `{ theme, resolvedTheme, toggleTheme, setTheme }` and renders `data-theme` on `<html>`. **`app/globals.css`** defines two full sets of CSS custom properties under `:root`/`html[data-theme="light"]` and `html[data-theme="dark"]` — every color, gradient, shadow uses `var(--...)` tokens. Theme persists to `localStorage` and respects `prefers-color-scheme` when `theme === "system"`. Pages consume the theme via `useContext(ThemeContext)` (e.g. `home/page.tsx` switches particle colors by `resolvedTheme`).

### Path alias

`@/*` maps to the project root (see `tsconfig.json` + jest `moduleNameMapper`). Always import via `@/app/...` not relative paths.

## Two different measurement-page implementations (known divergence)

This codebase has **two** measurement UIs that don't share logic:

1. **`app/measurement/[type]/page.tsx`** — the dynamic, intended entry point. Uses the `useMeasurementForm()` hook (form persistence via Zustand store, `ConfirmDialog`, `prepareSubmit`/`handleSubmit` split). Does **not** define its own measurements list inline — relies on the hook.
2. **`app/measurement/full-body-suit/page.tsx`** — a fuller standalone page. Has its own inline measurements list (16 fields, 5 groups), its own state, its own `localStorage` persistence (key `full-body-suit-measurements`), its own validation, and uses the `ConfirmDialog` component. Does **not** use the `useMeasurementForm` hook.

Be aware of this split when making changes to measurement behavior — fix may need to land in both places or be consolidated.

## Testing conventions

- **Jest** (`jest-environment-jsdom`, `next/jest`).
- **`jest.setup.js`** mocks `IntersectionObserver` and the entire `gsap` module — tests don't touch real DOM observation or GSAP timelines.
- The example test (`__tests__/components/LoadingSpinner.test.tsx`) also mocks `framer-motion`'s `motion.div` to a plain `div`. Follow this pattern: mock animation libraries in component tests.
- Coverage is collected from `app/**` excluding `.d.ts`, `_*.` (private), `layout.*`, and `api/**`.

## Styling notes

- **Tailwind v3** + the custom design system in `globals.css`. Prefer existing utility classes (`.glass`, `.card`, `.btn-primary`, `.btn-secondary`, `.input`, `.gradient-text`, `.badge`) over one-off Tailwind — they encode the visual identity (glassmorphism, gradient glow, noise texture overlay).
- `app/utils/ui.tsx` has reusable building-block components (`cn`, `AnimatedContainer`, `Badge`, `GlassCard`, `GradientText`, `Skeleton`) — import from here rather than re-creating.
- Colors are theme-variable-driven; never hardcode hex values in components (the one exception is the admin-panel code which uses raw Tailwind grays like `bg-gray-800` — that panel predates the theme system and doesn't use `var(--...)` tokens).

## Build config caveat

`next.config.ts` sets `eslint.ignoreDuringBuilds: true` AND `typescript.ignoreBuildErrors: true` — the production build passes even with lint/type errors. Run `npm run lint` and check `tsc` separately if you need to catch issues.
